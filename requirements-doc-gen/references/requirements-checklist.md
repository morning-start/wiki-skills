## 1. 文档规范性检查

### 1.1 基本信息完整性
- [ ] 文档标题明确（SRS 或 PRD）
- [ ] 文档版本号
- [ ] 编写日期
- [ ] 作者信息
- [ ] 目标读者说明
- [ ] 修订记录（版本、日期、修订内容、修订人）

### 1.2 语言规范性
- [ ] 避免使用模糊词汇（"可能"、"大概"、"应该"）
- [ ] 需求描述具体、可验证
- [ ] 术语定义清晰，全文一致
- [ ] 逻辑结构清晰，章节组织合理

### 1.3 可追溯性
- [ ] 每个功能需求有唯一编号（如 REQ-001）
- [ ] 每个需求有明确的优先级
- [ ] 需求与用例/设计/测试用例的对应关系

## 2. 功能需求完整性检查

### 2.1 基础信息
- [ ] 项目背景清晰，说明解决的问题
- [ ] 产品目标明确，包含可衡量的 KPI 指标
- [ ] 项目范围明确（包含什么、不包含什么）
- [ ] 目标用户画像清晰（角色、场景、痛点）

### 2.2 功能清单完整性
- [ ] 使用 MoSCoW 法则对功能分类（Must-have, Should-have, Could-have, Won't-have）
- [ ] 核心功能覆盖用户主要场景
- [ ] 功能优先级合理，版本规划清晰
- [ ] 每个功能有详细描述（输入、处理逻辑、输出、异常处理）

### 2.3 核心功能检查清单

#### 用户管理类
- [ ] 用户注册流程
- [ ] 用户登录流程（支持多种认证方式）
- [ ] 密码重置流程
- [ ] 账户激活流程
- [ ] 用户信息管理
- [ ] 权限管理（角色、权限配置）

#### 数据操作类
- [ ] 数据创建/新增
- [ ] 数据查询/搜索
- [ ] 数据修改/更新
- [ ] 数据删除
- [ ] 批量操作
- [ ] 数据导入/导出

#### 业务流程类
- [ ] 正常流程（Happy Path）
- [ ] 异常流程（Error Path）
- [ ] 边界条件处理
- [ ] 并发冲突处理
- [ ] 审批流程（如适用）

#### 交互反馈类
- [ ] 操作成功反馈
- [ ] 操作失败提示
- [ ] 加载状态提示
- [ ] 空状态提示
- [ ] 确认对话框（关键操作）

### 2.4 用户旅程完整性
- [ ] 新用户引导流程（Onboarding）
- [ ] 核心功能使用流程
- [ ] 异常情况处理流程
- [ ] 退出/取消流程

## 3. 非功能需求完整性检查

### 3.1 性能需求
- [ ] 响应时间要求（平均响应时间、P99 响应时间）
- [ ] 吞吐量要求（并发用户数、请求量）
- [ ] 可用性要求（SLA、停机时间限制）
- [ ] 启动时间要求（移动端应用启动时间）

### 3.2 安全需求
- [ ] 身份认证方式（密码、OAuth2.0、SSO、双因素认证）
- [ ] 权限控制模型（RBAC、ABAC）
- [ ] 数据加密要求（传输加密、存储加密）
- [ ] 安全防护措施（XSS、CSRF、SQL 注入、速率限制）
- [ ] 审计日志要求（记录哪些操作、保留多久）
- [ ] 合规性要求（GDPR、等保、行业法规）

### 3.3 可靠性需求
- [ ] 故障恢复时间（RTO）
- [ ] 数据恢复时间（RPO）
- [ ] 备份策略（全量备份、增量备份、保留周期）
- [ ] 灾备方案（异地备份、故障切换）
- [ ] 容错机制（重试、降级、熔断）

### 3.4 可维护性需求
- [ ] 代码规范要求
- [ ] 测试覆盖率要求
- [ ] 日志规范要求（日志格式、级别）
- [ ] 监控指标要求（性能监控、错误监控）
- [ ] 部署方式（容器化、CI/CD）

### 3.5 兼容性需求
- [ ] 浏览器兼容性（支持的浏览器和版本）
- [ ] 操作系统兼容性（Windows、macOS、Linux、iOS、Android）
- [ ] 设备兼容性（屏幕尺寸、分辨率）
- [ ] 第三方依赖版本要求

### 3.6 可用性需求
- [ ] 界面设计规范（颜色、字体、布局）
- [ ] 无障碍支持（屏幕阅读器、键盘导航、对比度）
- [ ] 国际化/多语言支持
- [ ] 新用户学习时间要求

## 4. 多端需求完整性检查

### 4.1 终端覆盖清单
- [ ] Web 端（支持哪些浏览器）
- [ ] iOS 端（支持 iOS 版本范围）
- [ ] Android 端（支持 Android 版本范围）
- [ ] 桌面端（Windows、macOS）
- [ ] 游戏端（主机、PC、移动端游戏）

### 4.2 多端差异说明
- [ ] 各端功能差异（哪些功能只在某端实现）
- [ ] 各端性能差异（不同平台的性能指标要求）
- [ ] 各端交互差异（触摸 vs 键鼠 vs 手柄）

### 4.3 体验一致性要求
- [ ] 核心功能各端保持一致
- [ ] 视觉风格统一（品牌色、字体、图标）
- [ ] 数据实时同步策略

### 4.4 数据同步策略
- [ ] 在线同步机制
- [ ] 离线缓存机制
- [ ] 冲突解决机制（时间戳优先、人工介入）
- [ ] 数据一致性保证

### 4.5 平台特性利用
- [ ] Web 端：响应式布局、浏览器特性
- [ ] 移动端：推送通知、定位、摄像头、传感器
- [ ] 桌面端：系统托盘、快捷键、多窗口
- [ ] 游戏端：手柄支持、成就系统、云存档

## 使用指南

### 如何使用本检查清单

1. **逐项检查**：按章节顺序逐项检查需求文档
2. **标注状态**：
   - [✓] 已完成且符合要求
   - [?] 不明确或需要补充
   - [✗] 缺失或不满足要求
3. **补充完善**：针对 [?] 和 [✗] 项，与用户交互补充信息
4. **迭代验证**：补充后重新检查，直至所有项为 [✓]

### 常见缺失项提醒

**高频遗漏项**：
- 非功能需求（性能、安全、可维护性）
- 多端需求差异说明
- 异常处理流程
- 数据备份与恢复策略
- 审计日志要求
- 第三方服务可用性假设

**质量提升建议**：
- 避免使用模糊词汇
- 需求描述必须可测试、可验证
- 明确多端体验一致性要求
- 补充用户旅程和交互流程
- 定义清晰的 KPI 指标
