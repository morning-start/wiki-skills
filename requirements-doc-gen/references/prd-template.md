## 2. 背景与目标

### 2.1 业务背景
> 描述产品要解决的痛点、市场机会

**现状问题**：
- 传统考勤方式效率低下，需要人工统计
- 移动办公场景下无法实时打卡
- 缺乏灵活的请假审批流程

**市场机会**：
- 远程办公趋势增长，在线考勤需求激增
- 竞品功能单一，缺乏智能化特性

### 2.2 产品目标
> 明确产品的商业目标和成功指标

**短期目标（3 个月）**：
- 完成核心功能开发（考勤打卡、请假审批）
- 获得首批 100 家企业试用
- 用户留存率达到 60%

**长期目标（1 年）**：
- 市场份额达到 20%
- 用户满意度评分 ≥ 4.5/5.0
- 实现盈亏平衡

| 指标 | 目标值 | 测量周期 |

| 注册转化率 | 15% | 月度 |

## 3. 目标用户画像

### 3.1 用户角色

#### 角色 1：企业员工
**基本信息**：
- 年龄：25-35 岁
- 职业：办公室职员、销售、技术人员
- 工作场景：办公室、出差途中、客户现场

**核心需求**：
- 快速打卡，节省时间
- 查看考勤记录，了解考勤状态
- 申请请假，流程简单明了

**痛点**：
- 经常忘记打卡
- 审批流程不透明
- 考勤统计不准确

**使用场景**：
> 早上 9:00，小王在地铁上打开 App，点击"一键打卡"，系统自动定位确认位置，打卡成功。下班前 10 分钟，系统提醒小王打卡。

#### 角色 2：HR 管理员
**基本信息**：
- 年龄：30-45 岁
- 职业：人事专员、HR 经理
- 工作场景：办公室、会议室

**核心需求**：
- 批量管理员工考勤数据
- 审批请假申请，提高效率
- 生成考勤报表，便于分析

**痛点**：
- 手工统计工作量大
- 数据准确性难以保证
- 缺乏数据可视化分析

**使用场景**：
> 每周一上午，HR 小李打开系统，一键导出上周考勤报表，发现 3 名员工考勤异常，系统自动发送提醒邮件。

## 5. 用户旅程与交互流程

### 5.1 核心流程

#### 流程 1：用户登录流程
```
开始
  │
  ├─> 输入邮箱/手机号
  │
  ├─> 输入密码
  │
  ├─> 点击"登录"按钮
  │
  ├─> [分支] 验证成功 → 跳转首页
  │
  └─> [分支] 验证失败 → 显示错误提示
          │
          ├─> 错误：邮箱不存在 → 提示"该邮箱未注册"
          ├─> 错误：密码错误 → 提示"密码错误"
          └─> 错误：账户锁定 → 提示"账户已锁定，请联系管理员"
```

#### 流程 2：打卡流程
```
开始
  │
  ├─> 点击"打卡"按钮
  │
  ├─> [分支] 位置验证
  │     ├─> 成功 → 获取打卡时间
  │     └─> 失败 → 显示"当前位置不在打卡范围内"
  │
  ├─> [分支] 时间验证
  │     ├─> 允许 → 打卡成功
  │     └─> 超时 → 显示"不在打卡时间内"
  │
  └─> 显示打卡成功/失败提示
```

### 5.2 交互说明

#### 交互 1：登录按钮点击
**触发条件**：用户点击登录按钮
**反馈内容**：
- 按钮显示加载状态（转圈）
- 验证通过：跳转首页
- 验证失败：在输入框下方显示错误提示（红色文字）

#### 交互 2：打卡成功
**触发条件**：用户打卡成功
**反馈内容**：
- 弹出成功提示（Toast，显示 2 秒）
- 播放成功音效
- 更新打卡状态为"已打卡"

### 5.3 异常状态处理

| 异常场景 | 错误提示 | 处理建议 |
|---------|---------|---------|
| 网络连接失败 | "网络连接失败，请检查网络设置" | 重试按钮 |
| 定位服务未开启 | "请开启定位权限" | 跳转设置页面 |
| 位置不在范围内 | "当前位置不在打卡范围内" | 查看打卡范围地图 |
| 账户已锁定 | "账户已锁定，请联系管理员" | 显示联系电话 |

## 6. 功能详细说明

### 6.1 用户注册

**功能描述**：新用户通过邮箱注册账号

**交互流程**：
1. 用户输入邮箱
2. 系统验证邮箱格式
3. 系统发送验证码到邮箱
4. 用户输入验证码
5. 用户设置密码
6. 注册成功，自动登录

**界面原型描述**：
- 页面标题："注册账号"
- 输入框 1：邮箱（placeholder: "请输入邮箱"）
- 输入框 2：验证码（右侧按钮"获取验证码"，点击后倒计时 60 秒）
- 输入框 3：密码（placeholder: "请设置密码"，显示/隐藏切换按钮）
- 按钮："注册"（主按钮）
- 链接："已有账号？立即登录"

**字段约束**：
- 邮箱：符合 RFC 5322 标准，长度 6-100 字符
- 验证码：6 位数字，有效期 5 分钟
- 密码：8-20 位，包含字母和数字

### 6.2 GPS 定位打卡

**功能描述**：用户通过 GPS 定位进行考勤打卡

**交互流程**：
1. 用户点击"打卡"按钮
2. 系统请求定位权限
3. 系统获取当前位置（精确到 50 米内）
4. 系统验证当前位置是否在打卡范围内
5. 验证通过，记录打卡时间和位置

**界面原型描述**：
- 卡片显示：当前状态（未打卡/已打卡/迟到/早退）
- 大按钮："一键打卡"（圆形按钮，图标为手指点击）
- 信息展示：
  - 当前时间：实时更新
  - 当前位置：自动获取，显示为"XX 省 XX 市 XX 区 XX 路 XX 号"
  - 打卡时间：上次打卡时间
- 地图预览：小窗口显示当前位置和打卡范围

**打卡范围配置**：
- 默认范围：以公司地址为中心，半径 500 米
- 支持配置：管理员可调整范围半径（100-2000 米）

## 7. 多端体验策略

### 7.1 终端覆盖

| 终端类型 | 支持版本 | UI/UX 策略 | 功能差异 |
|---------|---------|-----------|---------|
| Web 端 | Chrome 90+, Firefox 88+, Safari 14+ | 响应式布局，支持键盘快捷键 | 全功能 |
| iOS 端 | iOS 14+ | 原生体验，支持手势操作 | 支持桌面小组件 |
| Android 端 | Android 10+ | 原生体验，支持悬浮窗 | 支持桌面小组件 |
| 桌面端 | Windows 10/11, macOS Monterey+ | 独立窗口，支持系统托盘 | 不支持 GPS 打卡（仅 Wi-Fi） |

### 7.2 体验一致性

#### 一致性原则
- 核心功能各端保持一致（登录、打卡、请假）
- 视觉风格统一（品牌色、字体、图标）
- 数据实时同步

#### 差异化处理
- **Web 端**：适合复杂操作（报表配置、审批流程）
- **移动端**：适合快速操作（打卡、请假申请）
- **桌面端**：适合长时间使用（数据统计、导出）

### 7.3 数据同步策略

#### 在线场景
- 操作实时同步到服务端
- 各端实时接收更新推送

#### 离线场景
- 本地缓存最近 7 天数据
- 离线操作标记为"待同步"
- 联网后自动同步

#### 冲突解决
- 时间戳优先：取最新操作
- 人工介入：冲突数据标记，用户手动选择

## 8. 数据埋点需求

### 8.1 关键事件定义

| 事件名称 | 触发条件 | 属性参数 |
|---------|---------|---------|
| page_view | 页面加载 | page_name, user_id |
| click_login | 点击登录按钮 | login_method (邮箱/微信/手机) |
| login_success | 登录成功 | user_id, login_time |
| click_punch | 点击打卡按钮 | punch_type (上班/下班) |
| punch_success | 打卡成功 | user_id, punch_time, location |
| apply_leave | 提交请假申请 | user_id, leave_type, duration |
| export_report | 导出报表 | user_id, report_type, date_range |

### 8.2 留存分析
- 次日留存：用户注册后第 2 天是否有登录
- 7 日留存：用户注册后第 7 天是否有登录
- 30 日留存：用户注册后第 30 天是否有登录

### 8.3 转化漏斗
```
访问落地页 → 点击注册 → 输入邮箱 → 输入验证码 → 注册成功 → 首次登录 → 首次打卡 → 持续使用
  100%     →   80%   →   70%    →    60%    →    55%    →   50%    →   40%    →   25%
```

## 9. 非功能需求

### 9.1 性能需求
- 页面加载时间 ≤ 2 秒
- 接口响应时间 ≤ 500ms
- 首屏渲染时间 ≤ 1 秒

### 9.2 可用性需求
- 界面简洁易懂，新用户 5 分钟内上手
- 关键操作有明确提示和引导
- 支持键盘快捷键（Web 端和桌面端）

### 9.3 兼容性需求
- Web 端：支持主流浏览器最新 2 个版本
- 移动端：支持主流手机品牌（苹果、华为、小米、OPPO、vivo）
- 屏幕适配：支持 320px - 2560px 宽度

### 9.4 无障碍需求
- 符合 WCAG 2.1 AA 标准
- 支持屏幕阅读器
- 支持键盘导航
