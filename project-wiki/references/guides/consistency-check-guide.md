# 文档一致性检查指南

## 概述

本文档提供系统性的文档一致性检查方法，涵盖 7 大类一致性检查内容，帮助团队保障文档质量、降低理解偏差、提升协作效率。

### 核心目标

- **同一事实，在所有相关文档中表述相同**
- **逻辑自洽，需求 - 设计 - 实现 - 测试对齐**
- **版本同步，所有文档版本号、状态一致**

### 适用场景

- 文档生成后的质量检查
- 文档更新后的一致性验证
- 版本发布前的文档审核
- 项目里程碑评审

---

## 一、术语与命名一致性检查

### 1.1 技术术语一致性

**检查目标**：确保技术术语在全文中使用一致

**检查项**：

1. **术语定义检查**
   - [ ] 首次出现的术语是否有明确定义
   - [ ] 同一术语在不同文档中的定义是否一致
   - [ ] 是否存在同义词混用（如"模块"vs"组件"）

2. **术语使用检查**
   - [ ] 全文是否统一使用同一术语
   - [ ] 是否存在中英文混用（如"API"vs"接口"）
   - [ ] 专业术语是否符合行业标准

3. **缩写规范检查**
   - [ ] 首次出现是否注明全称（如"CI/CD（持续集成/持续交付）"）
   - [ ] 全文缩写是否统一
   - [ ] 缩写是否符合行业惯例

**检查工具**：
- 文本搜索工具（grep、VS Code 全局搜索）
- 术语管理工具（如 TBX、TermBase）
- 自定义脚本（检查术语频率和分布）

**示例**：
```markdown
# 错误示例 - 术语混用
文档 A: "系统采用微服务架构"
文档 B: "系统采用微服务架构风格"
文档 C: "系统采用微服务架构模式"

# 正确示例 - 术语统一
所有文档："系统采用微服务架构（Microservices Architecture）"
```

---

### 1.2 实体命名一致性

**检查目标**：数据库表名、接口名、类名等在各文档中一致

**检查项**：

1. **数据库命名**
   - [ ] 表名在 ER 图、数据字典、代码 ORM 映射中一致
   - [ ] 字段名在各文档中拼写一致
   - [ ] 命名规范统一（驼峰 vs 下划线）

2. **API 接口命名**
   - [ ] 接口名在 API 文档、代码注释、实现代码中一致
   - [ ] URL 路径命名规范统一
   - [ ] HTTP 方法使用一致

3. **代码实体命名**
   - [ ] 类名、方法名在文档和代码中一致
   - [ ] 变量命名在各处一致
   - [ ] 命名规范符合语言约定

**检查工具**：
- 代码分析工具（AST 解析）
- API 文档生成工具（Swagger/OpenAPI）
- 数据库文档工具

**示例**：
```markdown
# 错误示例 - 命名不一致
ER 图：user_table
数据字典：users
代码 ORM: User

# 正确示例 - 命名统一
ER 图：users
数据字典：users
代码 ORM: User (映射到 users 表)
```

## 二、内容逻辑一致性检查

### 2.1 需求↔设计↔实现↔测试对齐

**检查目标**：确保四个环节完全对齐，无遗漏、无偏差

**检查项**：

1. **需求 - 设计对齐**
   - [ ] 需求文档中每个功能点在设计文档中有对应方案
   - [ ] 设计方案覆盖所有需求
   - [ ] 设计扩展功能有明确说明

2. **设计 - 实现对齐**
   - [ ] 设计方案在代码中完全实现
   - [ ] 代码变更有对应的设计文档更新
   - [ ] 技术债务有明确记录

3. **实现 - 测试对齐**
   - [ ] 测试用例覆盖所有功能点
   - [ ] 测试场景反映实际使用场景
   - [ ] 测试结果可验证

4. **需求 - 测试对齐**
   - [ ] 测试用例可追溯到需求项
   - [ ] 验收标准与需求一致
   - [ ] 测试通过即需求满足

**检查工具**：
- 需求追踪矩阵（RTM）
- 测试覆盖率工具
- 文档对比工具

**示例 - 需求追踪矩阵**：
```markdown
| 需求 ID | 需求描述 | 设计章节 | 实现模块 | 测试用例 | 状态 |
|---------|---------|---------|---------|---------|------|
| REQ-001 | 用户注册 | 3.2.1 | UserController.register | TC-001, TC-002 | ✅ |
| REQ-002 | 用户登录 | 3.2.2 | UserController.login | TC-003, TC-004 | ✅ |
| REQ-003 | 密码重置 | 3.2.3 | UserController.resetPassword | TC-005 | 🔄 进行中 |
```

---

### 2.2 版本一致性检查

**检查目标**：所有文档标注的版本号、发布日期、迭代周期匹配

**检查项**：

1. **版本号一致性**
   - [ ] SKILL.md 中的 version 字段与 Git Tag 一致
   - [ ] CHANGELOG.md 中的最新版本号与 SKILL.md 一致
   - [ ] ROADMAP.md 中的"当前版本"与实际情况一致
   - [ ] README.md 中的版本号（如有）与其他文档一致

2. **日期一致性**
   - [ ] CHANGELOG 中的发布日期与实际发布日期一致
   - [ ] 文档元数据中的日期逻辑正确（创建≤更新≤发布）
   - [ ] 所有日期格式统一为 YYYY-MM-DD

3. **迭代周期一致性**
   - [ ] ROADMAP 中的迭代周期与实际开发计划一致
   - [ ] 版本发布记录完整
   - [ ] 迭代目标与实际交付一致

**检查工具**：
- Git 标签管理
- 版本管理脚本
- 日期格式验证工具

**示例**：
```markdown
# 错误示例 - 版本不一致
SKILL.md: version: 5.0.0
CHANGELOG.md: ## [4.0.0] - 2024-01-15
Git Tag: v4.5.0

# 正确示例 - 版本统一
SKILL.md: version: 5.0.0
CHANGELOG.md: ## [5.0.0] - 2024-02-20
Git Tag: v5.0.0
ROADMAP.md: 当前版本：v5.0.0
```

---

### 2.3 状态同步检查

**检查目标**：需求变更时，相关文档同步更新

**检查项**：

1. **变更影响分析**
   - [ ] 需求变更识别完整
   - [ ] 受影响文档识别完整
   - [ ] 变更影响范围评估准确

2. **文档同步更新**
   - [ ] 设计文档随需求变更同步更新
   - [ ] API 文档随接口变更同步更新
   - [ ] 测试用例随功能变更同步更新
   - [ ] 用户手册随产品变更同步更新

3. **变更记录完整性**
   - [ ] 变更原因记录清晰
   - [ ] 变更内容记录详细
   - [ ] 变更影响记录完整
   - [ ] 变更审批记录完整

**检查工具**：
- 变更管理系统
- 文档版本对比工具
- 影响分析矩阵

## 三、格式与结构一致性检查

### 3.1 文档模板统一性

**检查目标**：所有文档使用统一的模板格式

**检查项**：

1. **模板使用检查**
   - [ ] 文档类型与模板匹配
   - [ ] 模板章节完整
   - [ ] 模板占位符全部替换

2. **格式规范检查**
   - [ ] 标题层级一致（# → ## → ###）
   - [ ] 列表格式统一（- vs 1.）
   - [ ] 代码块语言标注正确
   - [ ] 表格格式规范

3. **元数据完整性**
   - [ ] 文档标题
   - [ ] 创建日期
   - [ ] 最后更新日期
   - [ ] 负责人
   - [ ] 文档状态

**检查工具**：
- Markdown lint 工具
- 模板验证脚本
- 元数据提取工具

---

### 3.2 章节结构一致性

**检查目标**：同类文档章节结构统一

**检查项**：

1. **结构顺序检查**
   - [ ] 章节顺序符合规范
   - [ ] 必需章节不缺失
   - [ ] 可选章节标注清晰

2. **章节编号检查**
   - [ ] 章节编号连续
   - [ ] 编号格式统一
   - [ ] 跨文档引用编号正确

**示例 - 标准文档结构**：
```markdown
# 文档标题

## 1. 概述
## 2. 架构设计
## 3. 功能说明
## 4. 接口文档
## 5. 部署指南
## 6. 附录

# 错误示例 - 结构混乱
# 概述
## 功能说明  # 缺少章节编号
# 3. 接口文档  # 编号不匹配
```

---

### 3.3 编号体系一致性

**检查目标**：需求编号、用例编号、接口编号跨文档可追溯

**检查项**：

1. **编号规则检查**
   - [ ] 编号格式统一（如 REQ-001、TC-001、API-001）
   - [ ] 编号唯一性
   - [ ] 编号连续性

2. **跨文档追溯检查**
   - [ ] 需求编号在设计文档中正确引用
   - [ ] 用例编号正确关联需求
   - [ ] 接口编号在调用处正确引用

**示例**：
```markdown
# 需求文档
REQ-001: 用户注册功能

# 设计文档
3.2.1 用户注册模块 (对应 REQ-001)

# 测试文档
TC-001: 验证用户注册功能 (对应 REQ-001)
TC-002: 验证注册邮箱格式 (对应 REQ-001)
```

## 四、数据与参数一致性检查

### 4.1 接口文档一致性

**检查目标**：接口文档与实际实现完全一致

**检查项**：

1. **字段定义检查**
   - [ ] 请求字段名、类型、必填性与实现一致
   - [ ] 响应字段名、类型、必填性与实现一致
   - [ ] 字段说明准确无歧义

2. **示例值检查**
   - [ ] 请求示例可实际使用
   - [ ] 响应示例与实际返回一致
   - [ ] 示例值符合业务规则

3. **错误码检查**
   - [ ] 错误码定义完整
   - [ ] 错误码在错误处理文档和 API 文档中统一
   - [ ] 错误码说明清晰

**检查工具**：
- API 测试工具（Postman、curl）
- OpenAPI/Swagger 验证
- 自动化接口测试

**示例**：
```markdown
# 错误示例 - 接口不一致
文档：POST /api/users { "name": "string", "email": "string" }
实现：POST /api/users { "username": "string", "email": "string", "required": ["username"] }

# 正确示例 - 接口一致
文档：POST /api/users
请求体：
{
  "username": "string",  // 必填，用户名
  "email": "string"     // 必填，邮箱
}
实现：完全匹配
```

---

### 4.2 配置参数一致性

**检查目标**：配置参数在各文档中一致

**检查项**：

1. **环境变量检查**
   - [ ] 环境变量名在部署文档、代码、运维手册中一致
   - [ ] 默认值一致
   - [ ] 必填/可选标注一致

2. **配置文件检查**
   - [ ] 配置项名称一致
   - [ ] 配置项类型一致
   - [ ] 配置项说明一致

**示例**：
```markdown
# 错误示例 - 配置不一致
部署文档：DATABASE_URL
代码：DB_CONNECTION_STRING
运维手册：DB_URL

# 正确示例 - 配置统一
所有文档：DATABASE_URL (必填，数据库连接字符串)
```

---

### 4.3 数据库字段一致性

**检查目标**：数据库字段在各处一致

**检查项**：

1. **字段定义检查**
   - [ ] 字段名在 ER 图、数据字典、代码中一致
   - [ ] 字段类型一致
   - [ ] 约束条件一致（NOT NULL、UNIQUE 等）

2. **关系定义检查**
   - [ ] 外键关系定义一致
   - [ ] 索引定义一致
   - [ ] 触发器、存储过程定义一致

---

## 五、图表与文字一致性检查

### 5.1 图表与正文一致性

**检查目标**：图表内容与正文描述一致

**检查项**：

1. **组件名称检查**
   - [ ] 架构图中的组件名称与正文一致
   - [ ] 流程图中的步骤名称与正文一致
   - [ ] 时序图中的对象名称与正文一致

2. **交互逻辑检查**
   - [ ] 图表中的交互流程与正文描述一致
   - [ ] 数据流向正确
   - [ ] 条件分支完整

**检查工具**：
- Mermaid 渲染验证
- 图表对比工具
- 人工评审

**示例**：
```markdown
# 错误示例 - 图表与文字不一致
正文："用户请求经过网关到达认证服务"
架构图：用户 → 负载均衡器 → 应用服务器

# 正确示例 - 图表与文字一致
正文："用户请求经过网关到达认证服务"
架构图：用户 → 网关 → 认证服务
```

---

**检查目标**：图表编号、标题格式统一

**检查项**：

1. **编号格式检查**

   - [ ] 跨文档引用编号正确

2. **标题格式检查**
   - [ ] 标题格式统一（如"图 3-2 用户登录流程"）

   - [ ] 图表有引用说明

### 5.3 示例代码时效性

**检查目标**：示例代码反映当前版本

**检查项**：

1. **代码版本检查**
   - [ ] 示例代码与当前版本一致

## 最佳实践

### 1. 使用需求追踪矩阵（RTM）

创建和维护需求追踪矩阵，确保端到端一致性：

```markdown
| 需求 ID | 需求描述 | 设计章节 | 实现模块 | 测试用例 | 状态 | 备注 |
|---------|---------|---------|---------|---------|------|------|
| REQ-001 | ... | 3.2.1 | UserController | TC-001~005 | ✅ | 已完成 |
```

### 2. 引入文档评审清单（Checklist）

每次文档更新后，使用检查清单进行交叉验证。

### 3. 利用工具辅助

- **Swagger/OpenAPI**: 同步 API 文档
- **Confluence**: 页面关系图
- **Git**: 版本管理文档
- **Markdown Lint**: 格式检查
- **自定义脚本**: 自动化一致性检查

### 4. 建立文档审核流程

```
文档编写 → 自审 → 同行评审 → 一致性检查 → 发布
```

### 5. 定期文档健康检查

每周/每月定期进行文档一致性检查，发现问题及时修复。

## 使用示例

### 示例 1：版本发布前检查

**场景**：准备发布 v5.0.0 版本

**步骤**：
1. 运行版本号一致性检查
2. 检查 CHANGELOG 完整性
3. 验证 ROADMAP 状态更新
4. 检查所有文档的日期格式
5. 生成检查报告

### 示例 2：需求变更后检查
